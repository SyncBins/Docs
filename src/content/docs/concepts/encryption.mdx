The plain-English version of the threat model. Every byte the server sees is ciphertext; the master key is derived from a 12-word phrase and never leaves your device. Here's the full envelope, the pairing handshake, and exactly what the server can and can't see.

## Primitives

SyncBins only uses well-vetted, NaCl-family primitives — there's no custom crypto. Every primitive in the table below ships in libsodium and is exposed through `libsodium-wrappers` on the web client.

| Use | Primitive | Library call |
|-----|-----------|--------------|
| Symmetric encryption (items, blobs) | `XChaCha20-Poly1305` | `crypto_aead_xchacha20poly1305_ietf_*` |
| Key wrapping (K_item → device) | `Curve25519 sealed box` | `crypto_box_seal` |
| Master-key derivation | `Argon2id` (t=3, m=64MB) | `crypto_pwhash` |
| Recovery phrase | `BIP39` 128-bit entropy | — |
| Pairing handshake | `X25519 + HMAC-SHA-256` | `crypto_kx_*` + `crypto_auth` |

## From 12 words to a master key

During onboarding we generate 128 bits of entropy and encode it as a BIP39 12-word phrase. The phrase is the only thing you have to back up.

```ts title="client/crypto/master.ts"
// 12-word phrase → master key (client-side, never sent to server)
const seed   = bip39ToSeed(phrase);                       // 64 bytes
const salt   = sha256("syncbins.master.v1");              // fixed, app-wide
const master = await argon2id(seed, salt, {
  t: 3, m: 64 * 1024, p: 1, len: 32,                      // 64 MB, ~700ms on laptop
});
```

The master key is used for **pairing** and **recovery only** — it does not directly encrypt items. Each device generates its own Curve25519 keypair on first run; the device's private key is itself sealed to the master key and stored locally.

<Aside type="tip" title="The PIN shortcut">
Re-deriving the master key from the phrase takes ~700ms — fine for setting up a new device, annoying every time the browser locks. Setting a PIN (6+ digits) wraps the master key locally with a separate Argon2id pass keyed on the PIN. Next unlock is an Argon2id round on a short string — sub-100ms. See [Quick-unlock PIN](/get-started/quick-unlock/).
</Aside>

```ts title="client/crypto/envelope.ts"
// Per-item encryption (TypeScript-ish pseudocode)
const K_item   = randomBytes(32);
const nonce    = randomBytes(24);
const aad      = utf8(`${bin_id}|${type}|${ts}`);
const ct       = xchacha20poly1305.encrypt(K_item, nonce, plaintext, aad);
const wrapped  = devices.map(d => sodium.crypto_box_seal(K_item, d.pubkey));
postItem({ id, bin_id, type, ts, nonce, ct, wrapped });
```

### Big items: blobs

Items larger than 64 KB (screenshots, audio, files) take a different path: the body is encrypted locally just like above, then the ciphertext is uploaded directly to your storage backend via a short-lived SAS URL. Only the metadata + a `blob_ref` go through the JSON API.

## What the server can actually see

This is the part worth dwelling on. Three columns: what's fully readable to the server, what's structural metadata, and what's fully opaque.

<Lens>
  <Fragment slot="ok">
    - Item ID (ULID)
    - Bin ID
    - Type (`password`, `link`, …)
    - Timestamp
    - Sender device ID
    - Ciphertext size
  </Fragment>
  <Fragment slot="mid">
    - Bin emoji + hue + capacity
    - Device name + glyph
    - Sync version counter
    - Pubkey of each device
    - Storage backend URL
  </Fragment>
  <Fragment slot="no">
    - Item payload (any field)
    - Bin name
    - Master key / recovery phrase
    - Device private keys
    - Item search index
    - Your PIN
  </Fragment>
</Lens>

<Aside type="caution" title="Metadata is not nothing">
The type label and bin tag are intentionally leaked so the server can render fanout decisions efficiently. If you're storing something where even the *shape* matters (e.g. "lots of passwords appearing at 3 AM"), that signal is observable to anyone with database access. Use a self-hosted box to remove that risk.
</Aside>

## Threat model summary

| Adversary | Can read items? | Notes |
|-----------|-----------------|-------|
| Network observer (TLS-on-the-wire) | No | HTTPS + items already encrypted at the app layer. |
| Compromised SyncBins host | No | Sees ciphertext + metadata only. Cannot fabricate items signed by another device. |
| Stolen unlocked device | Yes | Set a PIN; revoke the device remotely from another paired device. |
| Recovery phrase + stolen device | Yes | Full access. Keep the phrase offline; rotate after exposure. |
| Malicious paired device | Yes | It's paired — it has the master key. Revoke + rotate to evict. |
| Quantum adversary, future | No (for now) | Curve25519 is breakable post-Q. Re-encrypt rolling out with PQ KEM in 2027. |

## Key rotation & revocation

Revoking a device is a metadata flag. The device's pubkey is marked revoked; new items skip it in the wrapped-key bundle; the server refuses its bearer token. Existing items the revoked device downloaded before revocation are obviously still on that device — revocation prevents future access, not past.

For full rotation (e.g. you think the phrase leaked), generate a new phrase from the Encryption settings pane. Every item is re-wrapped to fresh device keys; old wrapped-key bundles are dropped. This is a background job — expect a few minutes on a large bin.
