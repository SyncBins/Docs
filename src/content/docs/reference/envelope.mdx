The server never sees item plaintext. It stores an **envelope**: ciphertext, nonce, and a **wrapped-key bundle** ‚Äî one sealed item key per device that should decrypt the row.

Types are defined in `App/shared/src/envelope.ts` and validated via `ItemEnvelopeSchema` in `shared/src/api.ts`.

## Primitives (recap)

| Step | Algorithm |
|------|-----------|
| Item body | XChaCha20-Poly1305 (`crypto_aead_xchacha20poly1305_ietf_*`) |
| Wrap `K_item` to each device | Curve25519 sealed box (`crypto_box_seal`) |
| Master key from phrase | Argon2id + BIP39 seed |

Full threat model: [How encryption works](/concepts/encryption/).

## Inline envelope (`payloadEnc`)

Stored as JSON in SQLite column `payload_enc`:

```ts
interface ItemEnvelope {
  ct: string;    // base64 ciphertext + poly1305 tag
  nonce: string; // base64, 24-byte XChaCha20 nonce
  keys: Record<DeviceId, string>; // base64 sealed K_item per device pubkey
}
```

### Encryption (client write)

1. `K_item = randomBytes(32)`
2. `nonce = randomBytes(24)`
3. Build AAD as UTF-8 bytes of `binId + "|" + type + "|" + ts` ‚Äî server validates these wire fields match AAD
4. `ct = AEAD_Encrypt(K_item, nonce, plaintextJSON, aad)`
5. For each non-revoked device: `keys[deviceId] = crypto_box_seal(K_item, devicePubkey)`
6. `POST /api/items` with `{ payloadEnc: { ct, nonce, keys }, ‚Ä¶ }`

### Decryption (client read)

1. `K_item = crypto_box_seal_open(keys[myDeviceId], myPubkey, myPrivkey)`
2. Reconstruct same AAD from row metadata
3. `plaintext = AEAD_Decrypt(K_item, nonce, ct, aad)`

If `keys` lacks an entry for this device, decryption fails ‚Äî run re-wrap from a device that still has access, or the item predates pairing and was never re-wrapped.

## Blob-backed items

Large payloads store bytes in object storage; the item row references them:

```ts
interface BlobEnvelope {
  blobRef: string;   // opaque name, e.g. ULID.bin
  nonce: string;     // blob encryption nonce
  keys: Record<DeviceId, string>; // K_blob wrapped like inline
}
```

Wire fields on `ItemRow`: `blobRef`, `blobNonce`, `blobKeys` ‚Äî with `payloadEnc: null`.

Flow:

1. Encrypt file bytes locally ‚Üí ciphertext blob
2. `POST /api/blobs/upload-url` ‚Üí presigned PUT
3. Upload ciphertext
4. `POST /api/items` with metadata payload (small JSON in `payloadEnc`) **or** metadata-only inline fields plus blob refs ‚Äî large media types use `blobRef` on the row and encrypted metadata in `payloadEnc` when present

Exactly **one** of inline `payloadEnc` body **or** blob path is enforced at the API layer.

Download: `GET /api/blobs/:ref/url` ‚Üí presigned GET ‚Üí decrypt with `blobNonce` + wrapped key.

## Wrapped-key bundle rules

- Every **non-revoked** paired device must appear in `keys` on write (server validates).
- New device joins ‚Üí existing clients **re-wrap** items they can decrypt (background job in web client).
- Revoked devices are omitted from new writes; old ciphertext may still exist on their disk.

## Base64 encoding

REST API uses **standard base64** in JSON (see [REST API conventions](/reference/rest/#conventions)). Internal crypto helpers may use base64url in browser storage ‚Äî convert at the HTTP boundary to match zod schemas.

## AAD binding

AAD is the UTF-8 string `binId|type|ts` (three wire fields joined by pipe characters).

If an attacker with DB access swaps ciphertext between rows without re-encrypting, authentication fails on decrypt. The server checks `binId`, `type`, and `ts` on the wire match what the client claims.
