One compose file, one DNS record, one `docker compose up -d`. At the end you'll have `https://box.yourdomain.com` running your own SyncBins with auto-renewed TLS and an Azure Blob backend.

<Aside type="tip" title="What you'll have when this is done">
A single-process container behind a Caddy sidecar that auto-provisions Let's Encrypt certificates. The container serves the React app, the REST API, and the WebSocket relay on one port. Persistent metadata lives in SQLite; blobs live in your Azure container.
</Aside>

## Prerequisites

- A VPS or homelab box you can SSH into. A $4/mo Hetzner CX11 is plenty.
- Docker and the `docker compose` plugin installed.
- A domain you can point at the box. A subdomain like `box.yourdomain.com` is fine.
- An Azure Storage Account + container (or skip ahead to [other backends](/self-hosting/storage/)).

## Get it running

<Steps>

1. **Point DNS at your box.** Create an A record for `box.yourdomain.com` that points to your VPS's public IPv4. If you have IPv6, add an AAAA record too. Caddy needs DNS to resolve correctly before it can fetch a certificate, so do this before the rest.

2. **Drop in the three files.** Make a folder on the box (anywhere — `/srv/syncbins` is conventional) and create these:

   <FileTree>
   - /srv/syncbins/
     - docker-compose.yml
     - Caddyfile
     - .env
     - data/ *(created on first run)*
   </FileTree>

   ```yaml title="docker-compose.yml"
   services:
     syncbins:
       image: ghcr.io/syncbins/app:latest
       restart: unless-stopped
       volumes:
         - ./data:/data
       environment:
         DOMAIN: ${DOMAIN}
         STORAGE_BACKEND: azure
         AZURE_STORAGE_CONNECTION_STRING: ${AZURE_CONN}
         AZURE_BLOB_CONTAINER: syncbins
         JWT_SECRET_FILE: /data/jwt.secret
         LISTEN: ":8080"
       expose: ["8080"]

     caddy:
       image: caddy:2
       restart: unless-stopped
       ports: ["443:443", "80:80"]
       volumes:
         - ./Caddyfile:/etc/caddy/Caddyfile
         - caddy_data:/data
         - caddy_config:/config

   volumes:
     caddy_data:
     caddy_config:
   ```

   ```caddy title="Caddyfile"
   {$DOMAIN} {
     reverse_proxy syncbins:8080
     encode zstd gzip
     header Strict-Transport-Security "max-age=31536000"
   }
   ```

   ```ini title=".env"
   DOMAIN=box.yourdomain.com
   AZURE_CONN=DefaultEndpointsProtocol=https;AccountName=…;AccountKey=…
   ```

   <Aside type="tip" title="Don't have Azure?">
   SyncBins supports four backends out of the box: **Azure Blob**, **S3**, **Cloudflare R2**, and **local disk**. The compose file above uses Azure. For the others, see [Choosing storage](/self-hosting/storage/) — the only difference is three environment variables.
   </Aside>

3. **Bring it up.**

   <Tabs syncKey="install-target">
     <TabItem label="Docker compose">
       ```sh
       docker compose up -d
       docker compose logs -f syncbins
       # …wait for "ready" — usually 5–10 seconds.
       ```
     </TabItem>
     <TabItem label="NixOS">
       ```nix
       # /etc/nixos/configuration.nix
       { ... }: {
         services.syncbins = {
           enable = true;
           domain = "box.yourdomain.com";
           storage = {
             backend = "local";
             path = "/var/lib/syncbins/blobs";
           };
         };
       }
       ```
     </TabItem>
     <TabItem label="Bare binary">
       ```sh
       curl -L https://github.com/SyncBins/App/releases/latest/download/syncbins-linux-amd64 \
         -o /usr/local/bin/syncbins
       chmod +x /usr/local/bin/syncbins
       syncbins serve --domain box.yourdomain.com --storage local
       ```
     </TabItem>
   </Tabs>

   Now open `https://box.yourdomain.com` in a browser. You'll land on the onboarding wizard — that's exactly the same flow described in [Get Started → Welcome](/get-started/welcome/).

4. **Bootstrap your first device.** A brand-new SyncBins server has no devices. The very first device to hit the onboarding flow becomes the *bootstrap device*: it generates the master key, the recovery phrase, and signs itself in. After that, every subsequent device gets paired through the handshake (see [How encryption works](/concepts/encryption/)).

   <Aside type="caution" title="Bootstrap is one-shot">
   Until a bootstrap device exists, anyone who can reach the URL could claim the server. Run the bootstrap flow yourself, on a device you trust, immediately after the first startup. If you need to redo it, run `syncbins reset-bootstrap` on the box.
   </Aside>

5. **Verify it's healthy.** Three quick checks:
   - Visit `https://box.yourdomain.com/api/syncbin/status` — should return `{ "ok": true, "configured": true }`.
   - In Settings → Server, the pulsing dot should be green and the latency under 100ms.
   - `docker compose logs syncbins` shows no `ERROR` lines.

</Steps>
