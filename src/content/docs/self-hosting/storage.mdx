SyncBins stores **metadata in SQLite** and **large ciphertext blobs** (screenshots, files, audio) in a pluggable backend. The server never sees plaintext — blobs are encrypted on your device before upload.

**Default for self-hosters: local disk.** No cloud account, no extra env vars beyond the [quick start](/self-hosting/quickstart/). Switch to R2 or Azure only if you want blobs off the VPS or need more durability than a single disk.

## TL;DR

| Backend | When to use | Extra env vars |
|---------|-------------|----------------|
| **Local disk** | Homelab, VPS, “just run it” | None (`STORAGE_KIND=local` is the default) |
| **Cloudflare R2** | Cheap cloud blobs, zero egress | `STORAGE_KIND=r2` + four `R2_*` vars |
| **Azure Blob** | Already on Azure | `STORAGE_KIND=azure` + connection string |

There is no Settings UI for this — configure at **deploy time** in `docker/.env`. See [Configuration reference](/self-hosting/config/) for every variable.

## At-a-glance

| Backend | Cost / GB-mo | Egress | Setup | Best for |
|---------|--------------|--------|-------|----------|
| **Local disk** | $0 (your hardware) | $0 | 1 min | Most self-hosters |
| **Cloudflare R2** | ~$0.015 | **$0** | 5 min | Cloud without egress surprise |
| **Azure Blob** | ~$0.018 (Hot) | Per-GB | 10 min | Azure shops |

Items under ~64 KB stay **inline in SQLite** — only larger payloads hit blob storage.

## Configuration per backend

<Tabs syncKey="storage-backend">
  <TabItem label="Local disk (default)">
    ```ini title="docker/.env"
    STORAGE_KIND=local
    LOCAL_BLOB_DIR=/data/blobs
    BLOB_URL_SECRET=…   # required in production — openssl rand -hex 32
    ```

    Blobs live as `{ulid}.bin` under `LOCAL_BLOB_DIR`. In Docker, bind-mount `./data:/data` so both SQLite and blobs survive container rebuilds.

    **You do not need R2, S3, or Azure** for a fully working self-hosted SyncBins. This is what the dev compose uses.
  </TabItem>

  <TabItem label="Cloudflare R2">
    ```ini title="docker/.env"
    STORAGE_KIND=r2
    R2_ACCOUNT_ID=your-cloudflare-account-id
    R2_ACCESS_KEY_ID=…
    R2_SECRET_ACCESS_KEY=…
    R2_BUCKET=syncbins
    BLOB_URL_SECRET=…
    ```

    Create a bucket in the R2 dashboard, generate an API token scoped to that bucket, paste the credentials into `.env`. The server uses presigned PUT/GET URLs — clients upload directly to R2; ciphertext only.

    Also add these variables to the `syncbins` service in `docker-compose.yml` if they are not already passed through from `.env`.
  </TabItem>

  <TabItem label="Azure Blob">
    ```ini title="docker/.env"
    STORAGE_KIND=azure
    AZURE_STORAGE_CONNECTION_STRING=DefaultEndpointsProtocol=https;AccountName=…;AccountKey=…
    AZURE_BLOB_CONTAINER=syncbins
    BLOB_URL_SECRET=…
    ```

    Use the **Hot** tier. Cool/Archive makes large item retrieval painfully slow.
  </TabItem>
</Tabs>

<Aside type="caution" title="AWS S3">
The server today ships drivers for **local**, **R2**, and **Azure**. Generic AWS S3 is not a separate `STORAGE_KIND` yet. R2 is S3-compatible and is the usual choice for homelabbers who want object storage without egress fees.
</Aside>

## How big a bucket?

Rough guide for ciphertext (already encrypted on device):

- **Screenshots:** 50 KB – 500 KB each
- **Voice memos:** ~0.5 MB per minute
- **Files:** whatever you drop in

A heavy user might add ~50 MB/week. Local disk or a small R2 bucket is plenty for personal use.

## Switching backends later

1. Stop the server.
2. Copy blobs (`cp -r`, `rclone copy`, or `aws s3 sync` depending on source/target).
3. Update `STORAGE_KIND` and credentials in `.env`.
4. Start the server.

SQLite keeps the same `blob_ref` names; the new backend must contain the same `{ulid}.bin` objects.
